Interview Prep

System design interviews are about demonstrating structured thinking, knowing tradeoffs, and communicating clearly. This section gives you a repeatable framework and walks through 10+ full interview questions.

The 6-Step Framework

Every system design interview should follow this structure. It shows discipline and ensures you don't miss anything important.

1

Clarify Requirements (3–5 min)

Ask questions before designing. What features are in scope? Who are the users? What scale? Read-heavy or write-heavy? What are the latency requirements? Never assume — confirm. Write the requirements on the whiteboard.

2

Estimate Scale (2–3 min)

Back-of-envelope calculations: DAU, requests per second, storage needed, bandwidth. This drives your architecture decisions. Mention if you're estimating QPS, storage, or memory. Don't skip this — it shows engineering sense.

3

High-Level Design (10 min)

Draw the big boxes: client → CDN → load balancer → API servers → cache → databases. Cover the main data flow. Get the interviewer's buy-in before diving deep. Don't detail every component yet.

4

Define the Data Model (5 min)

What are the main entities? What does each table/document look like? What's the primary key? What indexes do you need? What's your read/write pattern? This often reveals your database choice.

5

Deep Dive (15 min)

Pick 2–3 critical/interesting components and go deep. Likely prompted by the interviewer. This is where you show expertise: sharding strategy, caching layer, consistency model, specific algorithms. Discuss tradeoffs.

6

Address Bottlenecks & Failures (5 min)

What are the single points of failure? What happens if the database goes down? What's the hotspot risk? How do you handle a viral moment? Show you can think about operational concerns, not just happy-path design.

Back-of-Envelope Estimation

Interviewers want to see you estimate confidently. These numbers are your vocabulary:

Time Conversions

1 day = 86,400 sec ≈ 100K sec
1 month = 2.5M sec
1 year = 30M sec

Data Sizes

char = 1B · int = 4B · long = 8B
image ≈ 300KB · video/min ≈ 50MB
tweet ≈ 280B

QPS from DAU

If 100M DAU, each makes 5 req/day:
500M req/day ÷ 100K sec = 5000 QPS
Peak ≈ 2–3× = 10–15K QPS

Storage

Twitter: 100M tweets/day
= 100M × 280B ≈ 28GB/day
= ~10TB/year (just text)

Tip: Round aggressively. Say "approximately 10K QPS" not "9,847 QPS." Estimation is about order-of-magnitude thinking, not precision. Justify your assumptions out loud.

Do's & Don'ts

✅ Do

  • Ask clarifying questions first
  • Think out loud — share your reasoning
  • State tradeoffs explicitly
  • Start simple, then evolve
  • Acknowledge what you don't know
  • Draw a diagram (even rough)
  • Check in with interviewer
  • Mention failure scenarios

❌ Don't

  • Jump to coding immediately
  • Choose a DB without justification
  • Design everything in detail first
  • Ignore scale entirely
  • Claim everything is perfect
  • Forget about security
  • Use buzzwords without explaining them
  • Stay silent for too long

Design a URL Shortener (like bit.ly)
BeginnerClassic
+
Requirements

Functional: Given a long URL, return a short URL. Given a short URL, redirect to original. Optional: custom aliases, expiry.

Non-functional: 100M URLs/day, 10:1 read:write ratio, low latency redirect (<10ms), high availability.

Scale Estimation

Writes: 100M/day = ~1200 QPS. Reads: 1.2B/day = 12K QPS. Storage: 100M/day × 500B/URL × 10 years = ~500GB total. Manageable.

Key Design Decisions
  • Hash generation: Use MD5 or SHA256, take first 7 chars (62⁷ ≈ 3.5 trillion combinations). Collision? Append counter and retry.
  • Database: NoSQL (Cassandra or DynamoDB) — simple key-value lookups, horizontally scalable.
  • Caching: Redis cache for hot URLs (80% of reads are on 20% of URLs). Cache with TTL. Reduces DB load dramatically.
  • Redirect: 301 (permanent) = browser caches, less server load. 302 (temporary) = tracks every click, more analytics data.
  • Custom aliases: Allow user to specify shortcode. Check availability in DB first. Use read-your-own-writes consistency.
Data Model

Table: urls(short_code PK, original_url, user_id, created_at, expires_at). Index on original_url if we need reverse lookup (check if long URL already shortened).

Interview Tip

If asked about unique ID generation at scale: use a distributed ID generator (Twitter Snowflake, or pre-generate a pool of IDs). Avoid auto-increment across distributed nodes — they conflict.

Design Instagram (photo sharing)
IntermediatePopular
+
Core Features

Upload photos/videos. Follow users. View news feed. Like and comment. Search.

Key Architecture Decisions
  • Photo storage: Store photos in object storage (S3), not in the database. DB stores metadata (URL, size, user_id, timestamp).
  • CDN: Serve photos from CDN edge nodes. Users around the world get photos from the nearest server. Critical for latency.
  • News feed: Use fanout-on-write for most users (pre-computed feeds in Redis). Use fanout-on-read for users with huge followings (celebrities). Merge at read time.
  • Database: PostgreSQL for users, follows, comments. Cassandra for activity feeds (high write throughput). Both used at Instagram historically.
  • Search: Elasticsearch for hashtag/user search. Not the primary DB.
Bottlenecks to Address
  • Photo upload: async processing — return immediately, process in background (generate thumbnails via Kafka/Lambda)
  • Feed generation: pre-compute vs lazy. Use a hybrid based on follower count
  • Hot content: popular posts get cached aggressively in Redis with high TTL
Design WhatsApp (chat system)
IntermediateClassic
+
Requirements

Functional: 1:1 and group messaging, media (images, video), message delivery status (sent/delivered/read), online presence.

Scale: 2B users, 100B messages/day ≈ 1M QPS.

Key Design Decisions
  • WebSocket vs HTTP polling: Use persistent WebSocket connections for real-time bidirectional communication. HTTP polling is too slow and wasteful. Server can push messages to client instantly.
  • Chat servers: Each client maintains a WebSocket with a chat server. A session service maps user_id → chat_server. Message routing: sender → sender's chat server → look up recipient's chat server → push message.
  • Message storage: HBase (wide-column, high write throughput). Schema: (user_id, conversation_id, timestamp, message_id, content). Time-ordered within conversation.
  • Offline delivery: If recipient is offline, store message in a "pending" queue. On connection, push pending messages. Use message_id to avoid duplicates.
  • End-to-end encryption: Signal Protocol. Keys stored on device only. Server never sees plaintext. Mention this — shows security awareness.
  • Media: Upload to S3, send URL in message. CDN delivers media to recipient. Don't put binary blobs in message DB.
Group Chats

Group chat is fanout problem. For small groups (<100): send individual messages to each member's inbox. For large groups: use Pub/Sub — members subscribe to group topic in Kafka/Redis.

Design Netflix (video streaming)
AdvancedPopular
+
Scale

220M subscribers. 15% of global internet traffic. Need to serve 4K video to millions simultaneously with minimal buffering.

Key Architecture Decisions
  • Video storage & CDN: Raw videos on S3. Transcoded into multiple formats/resolutions (adaptive bitrate) and pushed to Netflix's Open Connect CDN. ISPs host CDN appliances in their networks — video comes from next door, not across the internet.
  • Adaptive Bitrate Streaming (ABR): Client requests chunks of video. If network is slow, it downgrades quality (720p → 480p). If fast, upgrades. Uses DASH or HLS protocol.
  • Video processing pipeline: Original video → Kafka → transcoding workers → multiple quality versions → CDN. Fully async, parallelized.
  • Recommendation engine: Apache Spark for batch ML. Feature store for real-time features. Recommendations precomputed nightly and cached.
  • Metadata service: Microservices for titles, search, user profiles. Cassandra for watch history (time-series, high write).
  • Chaos Engineering: Netflix invented Chaos Monkey — randomly kills production servers to test resilience. Your design should be resilient to this.
Design Uber (ride sharing)
AdvancedClassic
+
Core Problem

Match riders with nearby drivers in real-time. Track driver locations. Handle trip lifecycle. All at global scale, low latency.

Key Design Decisions
  • Location tracking: Drivers send GPS location every 5 seconds via WebSocket/HTTP. This is ~500K updates/sec for Uber's scale. Store in Redis (fast, ephemeral) not a DB. Use geospatial indexing (Redis GEOADD, GEORADIUS commands).
  • Driver matching: When rider requests: query Redis for drivers within radius (geospatial query). Filter by availability. Rank by estimated pickup time. This is the dispatch service — extremely latency-sensitive.
  • Geohashing: Divide the world into a grid of cells (geohash). Drivers in a cell are stored together. Querying nearby drivers = querying current cell + adjacent cells. Efficient and scalable.
  • Trip service: Manages trip lifecycle (requested → accepted → started → completed). Saga pattern for state transitions. Stores in PostgreSQL (ACID needed for financial aspects).
  • Surge pricing: Pub/Sub consumer on trip_requested events. Real-time supply/demand calculation per geohash. Updates pricing for that area.
  • Notifications: Push notifications (APNs/FCM) for match found, driver arriving, etc. Via dedicated notification service fed by Kafka events.
Interview Tip

Geohashing and geospatial queries are the key technical insight here. If you know Redis GEORADIUS and geohash concepts, say so — it shows depth.

Design Twitter (social media feed)
IntermediateClassic
+
The Core Problem

Users post tweets. Users follow each other. When you open Twitter, you see a feed of recent tweets from everyone you follow — in the right order, fast.

Key Design: Feed Generation
  • Naive approach: On feed load, query tweets from all followed users, merge, sort by time. Problem: if you follow 1000 people, that's 1000 DB queries merged in real-time. Too slow.
  • Fanout-on-write (push): When user posts, immediately push tweet_id to all followers' feed lists in Redis. Feed load = single Redis list read. Fast reads, slow writes for popular users (Lady Gaga's tweet fans out to 100M users).
  • Hybrid: Fanout-on-write for users with <1M followers. Fanout-on-read for celebrities. At read time, merge the user's pre-computed feed with real-time queries to celebrities they follow. Twitter uses this approach.
  • Data model: tweets(id, user_id, text, created_at, media_url). follows(follower_id, followee_id). Redis sorted set for feeds: key = user_id, score = timestamp, member = tweet_id.
Design a Web Crawler & Search Engine
AdvancedSenior
+
Crawler Design
  • Seed URLs → URL Frontier: Priority queue of URLs to crawl. Prioritize by PageRank, freshness, domain.
  • Distributed fetchers: Multiple workers fetch URLs in parallel. Respect robots.txt. Rate limit per domain (don't DDoS).
  • Deduplication: Bloom filter to skip already-crawled URLs. Hash content to skip duplicate pages (same content, different URL).
  • Storage: Raw HTML in object storage. Extracted links back to URL frontier. Extracted text to indexing pipeline.
Search Index
  • Inverted index: Maps word → list of (doc_id, positions). "apple" → [(doc1, [5,20]), (doc2, [3])]. Built by MapReduce over crawled pages.
  • Serving: Query → tokenize → look up each word in inverted index → intersect result sets → rank by relevance → return top K.
  • Ranking: PageRank (link graph), TF-IDF (term frequency), freshness, click-through rate, semantic similarity.
Design YouTube (video platform)
Intermediate
+
Upload Flow

Client uploads video → API server streams to raw storage (S3) → publishes "video uploaded" event to Kafka → Transcoding workers consume event → generate multiple resolutions → push to CDN → update video metadata DB to status=available → notify uploader.

Key Decisions
  • Resumable uploads: Large video files need resumable upload. Upload in chunks. Track progress server-side. Resume from last chunk on failure.
  • Transcoding: Computationally expensive, async. Encode to H.264/H.265 at 1080p, 720p, 480p, 360p. Generate thumbnails. All in parallel with multiple workers.
  • View count: Don't update DB on every view — too much write load. Batch updates every minute using Redis counter. Consistency can be eventual here.
  • Comments: Nested comments — use closure table or materialized path pattern. Or denormalize top-level and reply structure separately.
Design a Rate Limiter
BeginnerComponent
+
Design
  • Algorithm: Token bucket (most practical). Each user gets a bucket of tokens (e.g., 100 req/min). Each request consumes a token. Tokens refill at a fixed rate. Bucket size = max burst.
  • Storage: Redis. Key: rate_limit:{user_id}. Value: {tokens, last_refill_time}. Atomic update via Lua script (check + update = one atomic operation, prevents race conditions).
  • Response: If token available → consume token → allow request. Else → return 429 with Retry-After header = time until next token.
  • Distributed: Multiple API servers all share the same Redis instance. Rate limit enforced globally, not per-server.
  • Config: Rate limits per API key, per user tier. Store limits in a config DB (so you can change without deployment).
Design a Distributed Key-Value Store
AdvancedSenior
+
Design (like a mini DynamoDB)
  • Data partitioning: Consistent hashing ring. Keys distributed across nodes. Virtual nodes prevent uneven distribution. Adding/removing nodes only moves ~1/N keys.
  • Replication: Each key replicated to N nodes (next N clockwise on the ring). Tunable: N=3, W=2, R=2 for quorum (W+R > N = strong consistency).
  • Consistency: Choose: quorum (strong) or last-write-wins (eventual). Vector clocks for conflict detection. Application-level conflict resolution for true eventual consistency.
  • Failure handling: Gossip protocol for node health. If a node is down, use sloppy quorum + hinted handoff (another node temporarily holds the data, hands it off when original recovers).
  • Storage engine: SSTable + LSM Tree (Log-Structured Merge Tree). Writes go to in-memory memtable first (fast). Flushed to sorted SSTables on disk. Reads merge SSTables. Used by Cassandra, LevelDB, RocksDB.
