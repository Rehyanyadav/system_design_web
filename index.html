🧱 Module 1 · Beginner

Fundamentals

Before designing any system, you need solid mental models. This module builds the foundation — how data moves, what slows it down, and how systems fail. Don't skip this.

How the Internet Works

The internet is a network of networks. When you type google.com into a browser, a surprisingly complex series of events happens in milliseconds.

Core model: The internet is built on a client-server model. A client (your browser, mobile app) makes requests. A server processes them and sends back responses.

The Request Journey

Here's what happens when you visit a website:

  1. Browser checks its local cache for the IP of the domain
  2. If not cached, asks your ISP's DNS resolver
  3. DNS resolver walks the hierarchy to find the IP address
  4. Browser opens a TCP connection to that IP (port 80 for HTTP, 443 for HTTPS)
  5. TLS handshake if HTTPS (encryption negotiation)
  6. Browser sends an HTTP GET request
  7. Server processes request, sends back HTML
  8. Browser parses HTML, fetches CSS/JS, renders page
🎯 Interview Tip "Walk me through what happens when you type google.com" is a classic warm-up. Know the DNS → TCP → TLS → HTTP flow cold.

DNS Resolution

DNS (Domain Name System) translates human-readable domain names into IP addresses. Think of it as the internet's phonebook.

DNS Hierarchy

DNS is hierarchical, not centralized. This makes it fast and resilient:

Example: resolving "api.example.com"

The 4-step resolver chain

Root nameservers → know where .com lives
TLD nameservers (.com) → know where example.com's nameserver is
Authoritative nameservers → know the actual IP for api.example.com
Result is cached at every level (TTL controls how long)

Key DNS Record Types

A — maps domain to IPv4 address
AAAA — maps domain to IPv6 address
CNAME — alias for another domain (used for CDNs)
MX — mail server records
TXT — arbitrary text (used for verification)

Design insight: DNS can be used for simple load balancing (return multiple IPs, client picks one). This is called DNS round-robin but it's not truly load-aware.

HTTP & HTTPS

HTTP (HyperText Transfer Protocol) is the foundation of data communication on the web. It's a stateless, request-response protocol.

HTTP Methods

GET — retrieve data (safe, idempotent)
POST — create new resource (not idempotent)
PUT — replace resource (idempotent)
PATCH — partially update resource
DELETE — remove resource (idempotent)

Idempotent means calling the same operation multiple times produces the same result. GET, PUT, DELETE are idempotent. POST is not — calling it twice creates two resources.

HTTP Status Codes You Must Know

200 OK · 201 Created · 204 No Content
301 Moved Permanently · 302 Found (temporary redirect)
400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found · 409 Conflict · 429 Too Many Requests
500 Internal Server Error · 502 Bad Gateway · 503 Service Unavailable · 504 Gateway Timeout

HTTP Versions

HTTP/1.1

  • Persistent connections
  • Head-of-line blocking
  • One request at a time per connection

HTTP/2

  • Multiplexing (many requests, one conn)
  • Header compression (HPACK)
  • Server push

HTTPS = HTTP + TLS. TLS encrypts the connection so nobody between you and the server can read the data. Every system you design should use HTTPS.

TCP vs UDP

These are the two main transport protocols. Choosing between them is a classic systems tradeoff.

TCP — Transmission Control Protocol

  • Reliable delivery (ACKs)
  • Ordered packets
  • Error correction
  • Flow & congestion control
  • Higher latency (3-way handshake)
  • More overhead

Use for: web, email, file transfer, databases

UDP — User Datagram Protocol

  • Low latency
  • No connection setup
  • Broadcast support
  • No delivery guarantee
  • Packets can arrive out of order
  • No error correction

Use for: gaming, video streaming, DNS, VoIP

🎯 Interview Tip "Why would you use UDP for video streaming?" — Because a dropped frame is better than waiting for a retransmit. Users accept slight quality loss but not freezing. This is the tradeoff mindset you need.

Latency, Throughput & Bandwidth

These three are the most cited numbers in system design. Know the difference and typical values.

Definitions

Latency — time for a single request to complete (ms). Lower is better.
Throughput — how many requests/operations per second a system handles. Higher is better.
Bandwidth — maximum data transfer rate (Gbps). A ceiling, not a guarantee.

Numbers Every Engineer Should Know

Latency Hierarchy

L1 cache: ~0.5 ns · L2 cache: ~7 ns
RAM access: ~100 ns
SSD random read: ~150 µs
HDD random read: ~10 ms
Same datacenter roundtrip: ~0.5 ms
Cross-region (e.g., US → EU): ~75 ms
Cross-continental (US → Asia): ~150 ms

Key insight: Disk is ~20,000× slower than RAM. Network is ~1000× slower than RAM. This is why caching exists.

Little's Law

A fundamental theorem in queuing theory: L = λ × W

Where: L = number of items in system, λ = arrival rate, W = time each item spends in system. If throughput is 1000 req/s and average latency is 50ms, then 50 requests are "in flight" at any time.

Vertical vs Horizontal Scaling

When a system can't handle more traffic, you scale it. There are two directions.

Vertical Scaling (Scale Up)

  • Simple — no code changes
  • No distributed system complexity
  • Strong consistency
  • Has a hardware ceiling
  • Single point of failure
  • Expensive at high-end

Add more CPU/RAM/SSD to existing machine

Horizontal Scaling (Scale Out)

  • No theoretical ceiling
  • Fault tolerant (no single server)
  • Cost-effective at scale
  • Requires load balancing
  • Session/state management complexity
  • Distributed system problems

Add more machines, split the load

Industry practice: Most systems start vertical (easier), then switch to horizontal when they hit limits or need high availability. Large companies like Google, Facebook horizontally scale everything.

What is a Distributed System?

A distributed system is a collection of independent computers that appear to users as a single coherent system. The computers communicate over a network.

Why build distributed systems?

Three main reasons:

1. Scale — one machine can't handle Netflix's traffic
2. Reliability — if one server dies, others keep serving
3. Geography — serve users from the nearest datacenter

The core challenge: In a distributed system, things fail independently. Networks drop packets. Servers crash. Clocks drift. Your system must handle partial failure gracefully.

The 8 Fallacies of Distributed Computing

Peter Deutsch and James Gosling (Sun Microsystems) identified 8 assumptions developers incorrectly make about distributed systems. Know these — they explain 90% of distributed system bugs.

  1. The network is reliable — It isn't. Packets drop. Connections time out.
  2. Latency is zero — Every network call takes time. Design for it.
  3. Bandwidth is infinite — Bandwidth costs money and has limits.
  4. The network is secure — Always encrypt. Always validate.
  5. Topology doesn't change — Servers are added/removed constantly.
  6. There is one administrator — Many teams, many policies, many failures.
  7. Transport cost is zero — Serialization, network hops cost CPU and time.
  8. The network is homogeneous — Many different hardware, OS, languages.
🎯 Interview Tip When designing a system, always ask: "What happens when this network call fails?" If you don't have an answer, your design is incomplete. Interviewers at product companies specifically probe failure modes.

🔧 Module 2 · Beginner–Intermediate

Core Components

Every system is built from the same set of components. Master these building blocks and you can design any system. We cover what each is, when to use it, and how companies use them in practice.

Databases

Databases are the most important component to understand for system design. Almost every interview decision flows back to a database choice.

Core question: Which database you pick depends on your data model, access patterns, scale requirements, and consistency needs. There is no universally "best" database.

SQL vs NoSQL

SQL (Relational)

  • Strong ACID guarantees
  • Powerful JOIN queries
  • Schema enforced — data integrity
  • Mature tooling, well understood
  • Harder to scale horizontally
  • Schema changes require migrations
  • Not great for unstructured data

Examples: PostgreSQL, MySQL, SQLite

NoSQL

  • Horizontal scaling built-in
  • Flexible schema
  • Optimized for specific access patterns
  • High write throughput
  • Eventual consistency (usually)
  • No JOIN support (in most)
  • Less mature than SQL

Examples: MongoDB, Cassandra, DynamoDB, Redis

NoSQL Types — Know These

Document

MongoDB, Firestore, CouchDB

JSON-like documents. Great for flexible schemas. Use for: catalogs, user profiles, content management.

Key-Value

Redis, DynamoDB, Riak

Fast O(1) lookups by key. Great for caching, sessions. Limited querying ability.

Wide-Column

Cassandra, HBase, BigTable

Rows with dynamic columns. Excellent write throughput. Great for time-series, IoT, analytics at scale.

Graph

Neo4j, Amazon Neptune

Nodes and edges. Perfect for relationship-heavy data: social networks, fraud detection, recommendations.

Time-Series

InfluxDB, TimescaleDB

Optimized for timestamped data. High ingest rates, efficient range queries. Use for metrics, monitoring.

Search

Elasticsearch, Solr

Full-text search, faceting, relevance ranking. Not a primary database — use alongside main DB.

🎯 Interview Tip "Which database would you use?" is the most common follow-up. Answer: describe your data model, access patterns, then justify your choice. Never just say "I'd use MongoDB" without reasoning.

ACID Properties & Transactions

ACID is what makes relational databases trustworthy for financial and transactional data.

Atomicity — All operations in a transaction succeed, or all fail. No partial updates. (e.g., bank transfer deducts AND credits, or neither happens.)

Consistency — A transaction brings the database from one valid state to another. Constraints are never violated.

Isolation — Concurrent transactions don't interfere. Results look as if transactions ran sequentially.

Durability — Committed transactions survive crashes. Written to disk, not just memory.

Real Example · Instagram

How Instagram evolved its database

Instagram started on PostgreSQL for user accounts and photos (relational, ACID). As they scaled, they added Cassandra for activity feeds (high write throughput, eventually consistent is OK). They use both — right tool for the job.

Caching

Caching stores copies of frequently accessed data in fast storage (usually RAM) to reduce load on slower storage (databases, APIs).

The 80/20 rule of caching: 20% of your data is accessed 80% of the time. Cache that 20% and you eliminate 80% of database load.

Cache Strategies

Cache-Aside (Lazy Loading)

Application checks cache first. On miss, loads from DB and populates cache.

Read: Check cache → miss → read from DB → write to cache → return
Write: Write to DB → invalidate/update cache

✅ Only caches what's needed. ❌ Cache miss causes 3 steps (slow first request).

Write-Through

Every write goes to cache AND database simultaneously.

Write: Write to cache + DB atomically → return
Read: Check cache → always a hit (if item exists)

✅ Cache always fresh. ❌ Write latency higher (two writes per operation).

Write-Back (Write-Behind)

Write to cache only. Async batch write to DB later.

Write: Write to cache → return (DB write async, batched)
Risk: Cache failure before DB write = data loss

✅ Very fast writes. ❌ Risk of data loss if cache crashes before flush.

Cache Eviction Policies

LRU (Least Recently Used) — evict what hasn't been accessed longest. Most common.
LFU (Least Frequently Used) — evict what's accessed fewest times.
TTL (Time To Live) — expire entries after a fixed time. Simple, predictable.

Redis

  • Rich data structures (sets, sorted sets, lists)
  • Pub/Sub messaging
  • Persistence options
  • Lua scripting

Use for: sessions, leaderboards, rate limiting, pub/sub

Memcached

  • Simpler, slightly faster
  • Multi-threaded
  • Only key-value (strings)
  • No persistence

Use for: simple object caching only

In most modern systems, Redis wins unless you have a very specific reason to use Memcached.

CDN — Content Delivery Networks

A CDN is a globally distributed network of servers (edge nodes) that cache content close to users. Instead of everyone hitting your origin server, they hit a nearby edge server.

Rule of thumb: Anything static (images, videos, CSS, JS bundles, PDFs) should be served via CDN. Dynamic content may also be cached with the right headers.

How CDNs Work

  1. User requests cdn.example.com/image.jpg
  2. DNS resolves to the nearest CDN edge server (using Anycast routing)
  3. Edge server checks its cache. Hit → return immediately. Miss → fetch from origin
  4. Edge caches the response and returns it to user
  5. Next request for same content from same region → instant cache hit
Real Example · Netflix

Netflix's Open Connect CDN

Netflix built their own CDN (Open Connect) deployed inside ISP data centers. Popular movies are pre-loaded to edge servers before peak hours. 95%+ of Netflix traffic is served from edge without hitting origin. They serve ~15% of global internet traffic this way.

Message Queues

Message queues enable asynchronous communication between services. Producer writes a message; consumer reads and processes it — they don't need to be running simultaneously.

Why queues? Decouple services. Handle traffic spikes (queue absorbs bursts). Enable retry on failure. Allow independent scaling of producers vs consumers.

Core Concepts

Producer — sends messages to the queue
Consumer — reads and processes messages
Topic/Queue — named channel for messages
Acknowledgment — consumer signals message was processed
Dead Letter Queue (DLQ) — messages that fail after N retries go here

Pub/Sub vs Point-to-Point

Point-to-Point: One producer → one consumer. Message deleted after consumed. Use for: task queues, job processing.

Pub/Sub: One producer → many consumers (subscribers). Each subscriber gets a copy. Use for: event broadcasting, notifications.

Kafka — Event Streaming

Apache Kafka is not just a queue — it's a distributed event log. Messages are retained for a configurable time and consumers read at their own pace.

Use Kafka when:

  • Need to replay events
  • Multiple consumers of same stream
  • High throughput (millions/sec)
  • Event sourcing architecture

Use RabbitMQ/SQS when:

  • Simple task queue (one consumer)
  • Need routing logic
  • Managed service (SQS)
  • Don't need replay
Real Example · Uber

Uber's dispatch with Kafka

When a rider requests a trip, Uber publishes a trip_requested event to Kafka. Multiple consumers read this: dispatch service (find a driver), surge pricing service (recalculate prices), analytics service (log the event). Each service works independently, at its own pace, without blocking the rider's request.

API Design

APIs are the contracts between services. Good API design is what separates maintainable systems from spaghetti.

REST vs GraphQL vs gRPC

REST

HTTP + JSON

Resource-based. Stateless. Simple. Universal. Over-fetching/under-fetching is the main problem.

GraphQL

HTTP + custom query lang

Client specifies exactly what data it needs. Great for mobile apps. Complex to implement server-side.

gRPC

HTTP/2 + Protocol Buffers

Fast binary protocol. Strongly typed. Great for internal service-to-service. Not browser-friendly natively.

Load Balancers

A load balancer distributes incoming traffic across multiple servers. Without it, horizontal scaling doesn't work.

Load Balancing Algorithms

Round Robin — requests distributed equally in order. Simple, works when servers are identical.
Least Connections — routes to server with fewest active connections. Better for variable request times.
IP Hash — same client IP always routes to same server. Useful for session stickiness.
Weighted Round Robin — servers with more capacity get more requests.

L4 vs L7 Load Balancers

L4 (Transport Layer) — routes based on IP/port only. Very fast. No visibility into content.
L7 (Application Layer) — routes based on HTTP headers, URL, cookies. Slower but smarter. Can route /api/* to API servers and /static/* to static servers.

🎯 Interview Tip Don't forget: the load balancer itself can be a single point of failure. At scale, you'd have multiple load balancers with a VIP (Virtual IP) handled by DNS or an upstream layer (e.g., AWS's ELB is already HA).

⚡ Module 3 · Intermediate

Design Patterns

This is where it gets interesting. Patterns are the solutions to problems that appear in every system. Understanding them — and when to use them — is what interviewers are testing.

Sharding (Horizontal Partitioning)

Sharding splits a large dataset across multiple database nodes (shards). Each shard holds a subset of the data. When no single database can hold all your data or handle all your queries, you shard.

Key question: What is the shard key? The shard key determines how data is distributed. A bad shard key creates hotspots (all traffic goes to one shard).

Sharding Strategies

Hash-Based Sharding

Hash the shard key and take modulo number of shards. Even distribution. Problem: adding a shard requires rehashing everything.

Range-Based Sharding

Split by key ranges (A–M on shard 1, N–Z on shard 2). Good for range queries. Risk of hotspots if data isn't uniformly distributed.

Directory-Based Sharding

Lookup table maps keys to shards. Very flexible. Single lookup table becomes a bottleneck and single point of failure.

Geographic Sharding

Data partitioned by region (US shard, EU shard, APAC shard). Great for data residency, latency. Imbalanced if regions have different loads.

Sharding gotcha: Cross-shard queries (JOINs across shards) are expensive or impossible. You must design your access patterns so most queries hit a single shard.

Consistent Hashing

Regular hash-based sharding has a problem: when you add or remove a node, you have to remap almost all keys. Consistent hashing solves this.

How it works

Imagine a ring of hash values from 0 to 2³². Servers and keys are both placed on this ring by hashing their identifiers. A key belongs to the first server clockwise from it on the ring.

When you add a server: only the keys between the new server and its predecessor move. That's ~1/N of total keys, not all of them.

When you remove a server: only its keys move to the next server clockwise. Again, ~1/N keys affected.

Real World Use

Where consistent hashing is used

Amazon DynamoDB, Apache Cassandra, Vimeo's memcache layer, Discord's server routing. It's the standard approach for distributing load across a dynamic set of nodes.

Virtual nodes: To prevent load imbalance (some servers getting more of the ring than others), each physical server is assigned multiple virtual positions on the ring. Cassandra uses 256 virtual nodes per server by default.

Replication

Replication keeps copies of data on multiple nodes. This provides fault tolerance (if one dies, others serve) and read scaling (distribute reads across replicas).

Leader-Follower (Master-Replica)

One leader accepts all writes. Followers replicate and handle reads. Most common pattern.

✅ Simple, strong consistency for writes ❌ Write bottleneck is the leader

Multi-Leader

Multiple leaders accept writes (often one per datacenter). Great for geo-distributed systems. Hard problem: write conflicts between leaders.

Leaderless (Quorum-Based)

Any node can accept writes. Reads/writes require a quorum (W + R > N where N=replicas). Used by Cassandra and DynamoDB.

🎯 Interview Tip "If your leader goes down, how does the system recover?" — Know the answer: automatic leader election (Raft/Paxos), or manual failover. In practice: PostgreSQL uses pg_ha or Patroni; MySQL uses Orchestrator. Cloud managed DBs handle it for you.

CAP Theorem

CAP Theorem states that a distributed system can only guarantee two of three properties simultaneously:

C

Consistency

Every read gets the most recent write, or an error

A

Availability

Every request gets a (possibly stale) response — no errors

P

Partition Tolerance

System continues despite network partition between nodes

The real choice: In a real distributed system, network partitions WILL happen. So you can't give up P. The real decision is: during a partition, do you prefer Consistency (reject requests) or Availability (serve potentially stale data)?

CP systems (choose consistency): HBase, Zookeeper, etcd. Stop serving or return errors during partition.
AP systems (choose availability): Cassandra, DynamoDB, CouchDB. Serve stale data during partition, reconcile after.

Real Example · Amazon DynamoDB

The AP choice — and why Amazon made it

DynamoDB is AP: it will serve reads even if they might be stale. Amazon's reasoning: a customer seeing a slightly stale shopping cart is far less bad than their cart being unavailable. For financial transactions, they add strongly consistent reads at higher cost. Context determines the right tradeoff.

ACID vs BASE

These are two philosophical approaches to database design that map to SQL and NoSQL respectively.

ACID

  • Strong consistency
  • Predictable behavior
  • Harder to scale
  • Higher latency under load

PostgreSQL, MySQL, Oracle

BASE

Basically Available, Soft state, Eventually consistent

  • Highly scalable
  • High availability
  • Eventual (not immediate) consistency
  • App must handle stale data

Cassandra, MongoDB (default), DynamoDB

Rate Limiting

Rate limiting prevents any single user or service from overwhelming your system. It's essential for public APIs and high-scale services.

Algorithms

Token Bucket

Bucket fills with tokens at a fixed rate. Each request consumes a token. Allows bursts up to bucket size. Used by AWS, Stripe.

Leaky Bucket

Requests enter a queue (bucket) and are processed at a fixed rate. Smooths out bursts. If bucket is full, requests are dropped.

Fixed Window Counter

Count requests in a fixed time window (e.g., 100 req/minute). Simple but has edge case: 200 requests at window boundary.

Sliding Window Log

Track timestamps of requests. Remove old timestamps. Count remaining. Accurate but memory-intensive at scale.

Implementation: Store rate limit state in Redis (per user key). Token bucket is most popular in practice. Return 429 Too Many Requests with Retry-After header when limit is hit.

Circuit Breaker Pattern

When a service is down or slow, repeatedly trying to call it wastes resources and creates cascading failures. The circuit breaker stops calls to a failing service and allows it to recover.

Three States

Closed — normal operation, all requests pass through. Count failures.
Open — failure threshold exceeded. All requests fail fast (no call made to downstream). A timer starts.
Half-Open — timer expires. Allow one test request. Success → close the circuit. Failure → stay open.

Real Example · Netflix Hystrix

Netflix's circuit breaker in practice

Netflix wraps every service call in a circuit breaker (Hystrix library). If the recommendation service is down, the circuit opens and Netflix serves a degraded response (top 10 popular movies) instead of an error. Users see something rather than nothing — this is graceful degradation.

The Fanout Problem

Fanout is when one write operation causes many downstream operations. Classic example: when a celebrity posts on Twitter, their tweet must be delivered to 100M followers.

Fanout on Write (Push)

  • Fast reads (pre-computed feeds)
  • Slow writes for popular users
  • Storage cost (100M copies of tweet)

Used for: most users with moderate followers

Fanout on Read (Pull)

  • Fast writes (one write)
  • Slow reads (aggregate at read time)
  • Higher read-time compute

Used for: celebrities/users with huge followings

Twitter's hybrid approach: Push model for normal users. Pull model for users with >1M followers (Lady Gaga, Obama). At read time, merge the push feed with pull from celebrities. This is a classic interview answer.

Bloom Filters

A Bloom filter is a probabilistic data structure that answers "is this element in the set?" with zero false negatives but possible false positives. Uses very little memory.

Use cases: check if a username is taken before querying the database. Check if a URL has been crawled. Check if a password is in a breach list. The Bloom filter gives a fast "definitely not" or "probably yes" answer.

Rule: Bloom filter says NO → definitely not in the set (no DB query needed). Bloom filter says YES → probably in set, but query DB to confirm. This eliminates most unnecessary DB queries.

🚀 Module 4 · Advanced

Advanced Distributed Systems

This module is for senior and staff-level interviews, or anyone designing large-scale distributed systems. These patterns appear at companies like Google, Netflix, Amazon, and Uber.

Microservices vs Monolith

A monolith is a single deployable unit with all functionality. Microservices split functionality into independent services that communicate over a network.

Monolith

  • Simple to develop & deploy initially
  • Easy debugging (single process)
  • No network overhead between components
  • Scales as one unit (can't scale just one part)
  • Deploy everything for any change
  • Tech stack locked for whole app

Microservices

  • Independent scaling of each service
  • Independent deployment
  • Fault isolation
  • Network complexity between services
  • Distributed tracing needed
  • Operational overhead (many services)
When to use microservices: When different components have very different scaling needs, different teams own different parts, or you need independent deployment velocity. Netflix, Amazon, Uber all moved to microservices after starting as monoliths.
Martin Fowler's rule: "Don't start with microservices." Start with a modular monolith. Extract services when you have a clear reason. Premature microservices add complexity without benefit.

API Gateway

An API gateway is the single entry point for all clients. It routes requests to the appropriate microservice, handling cross-cutting concerns.

What an API gateway does:

  • Request routing to microservices
  • Authentication & authorization (verify JWT token once, not in every service)
  • Rate limiting
  • SSL termination
  • Request/response transformation
  • Load balancing
  • Logging and metrics
Real World

Common API Gateways

AWS API Gateway, Kong, NGINX, Traefik, Envoy (at the service level). Netflix uses Zuul (open-sourced) as their API gateway. If you're on AWS, API Gateway + Lambda is a serverless alternative.

Service Mesh

In a microservices system with hundreds of services, handling communication (retries, timeouts, circuit breaking, mTLS, observability) in each service is hard. A service mesh moves this to the infrastructure layer.

A sidecar proxy (like Envoy) is injected alongside each service container. All network traffic goes through the sidecar. The mesh control plane configures all sidecars centrally.

What you get: Automatic mTLS between services, distributed tracing, circuit breaking, retries, traffic management — all without changing application code.

Popular options: Istio, Linkerd, Consul Connect, AWS App Mesh.

Event Sourcing

Instead of storing current state, event sourcing stores the sequence of events that led to the current state. The current state is derived by replaying events.

Bank account analogy: Normal DB stores current balance ($500). Event sourcing stores: deposited $1000, withdrew $300, deposited $200, withdrew $400. Current state = replay of events.

Benefits:

  • Full audit log baked in by design
  • Reconstruct state at any point in time ("what was the balance on March 15?")
  • Temporal decoupling — events can be processed by multiple consumers asynchronously
  • Easy to add new projections (new views of data from same events)

Challenges: Event schema evolution (old events with new code). Eventually consistent read models. Higher complexity. Storage grows over time (use snapshots to optimize).

CQRS — Command Query Responsibility Segregation

CQRS separates the read model from the write model. Write operations (commands) go to one model optimized for writes. Read operations (queries) go to a different model optimized for reads.

Write side: POST /order → Command Handler → Write DB (normalized) → emit event
Event: OrderCreated → Event Bus (Kafka) → Read Model Updater
Read side: GET /orders → Query Handler → Read DB (denormalized, fast)

The read model is often denormalized (pre-joined) for fast queries. It's eventually consistent with the write model.

Common pairing: CQRS + Event Sourcing. Commands create events. Events rebuild the read model. Used by financial systems, e-commerce orders, booking systems.

Saga Pattern — Distributed Transactions

In microservices, you can't use a single database transaction spanning multiple services. The Saga pattern solves distributed transactions through a sequence of local transactions with compensating actions on failure.

Choreography-based Saga

Each service publishes events. Other services subscribe and react. No central coordinator.

✅ Decoupled ❌ Hard to follow the flow, complex to debug

Orchestration-based Saga

A central orchestrator tells each service what to do. If a step fails, the orchestrator triggers compensating transactions to undo previous steps.

✅ Clear flow, easy to monitor ❌ Orchestrator becomes a bottleneck/single point

Real Example · E-commerce Order

Order placement as a saga

1. Order Service → creates order (local txn)
2. Payment Service → charges card
3. Inventory Service → reserves items
4. Shipping Service → creates shipment

If step 3 fails (out of stock): compensating txns run — refund payment, cancel order. Each service handles rollback of its own step.

Consensus — Raft Algorithm

How do distributed nodes agree on a value when any of them can fail? This is the consensus problem. Raft is the most understandable consensus algorithm (designed to be more understandable than Paxos).

Raft in a nutshell

Leader Election: One node is elected leader. Followers send votes. Majority wins.
Log Replication: All writes go to leader. Leader replicates to followers. Committed when majority acknowledge.
Safety: A new leader is always up-to-date (has all committed entries).

Used by: etcd (Kubernetes), CockroachDB, TiKV, Consul, InfluxDB clusters.

Distributed Locks

When multiple instances of a service need to coordinate access to a shared resource, distributed locks ensure only one instance acts at a time.

Using Redis: SET resource_lock unique_id NX PX 30000 — Set only if Not eXists, expire in 30 seconds. Release by checking the value matches and then deleting (atomic with Lua script).

Redlock: Redis's multi-node distributed lock algorithm. Acquires lock on majority of N Redis nodes. More resilient than single-node.

Caution: Distributed locks are tricky. Clock skew, GC pauses, and network delays can all cause a lock to expire while the holder is still working. For critical sections, consider database-level locking or stronger consensus (Zookeeper ZNodes).

Geo-Distribution & Multi-Region

Serving users globally requires placing infrastructure in multiple regions. This introduces the hardest distributed systems challenges.

Active-Passive: One region handles all writes. Others are read replicas. Simple but still requires cross-region writes for some users. Failover takes time (RPO/RTO).

Active-Active: All regions accept writes. Data must be replicated between regions. Conflict resolution is hard. Used by: DynamoDB Global Tables, CockroachDB, Spanner.

Real Example · Google Spanner

Globally distributed SQL with external consistency

Spanner achieves external consistency (stronger than serializability) across global datacenters using atomic clocks and GPS receivers (TrueTime API). Transactions are timestamped with guaranteed accuracy bounds. This lets Spanner wait out clock uncertainty and provide correct global ordering — something previously thought impossible with SQL at scale.

Observability — Metrics, Logs, Traces

In distributed systems, you can't just SSH into one machine to debug. Observability is the ability to understand the system's internal state from its external outputs.

Metrics — Numeric data over time. CPU usage, request rate, error rate, latency percentiles. Stored in time-series DB (Prometheus, InfluxDB). Visualized in Grafana.

Logs — Structured events from each service. Aggregated centrally (ELK stack, Splunk, Datadog). Good for debugging specific errors.

Traces — Track a single request as it flows through multiple services. Shows exactly where time was spent and where errors occurred. Tools: Jaeger, Zipkin, AWS X-Ray, Datadog APM.

The three pillars: Metrics tell you something is wrong. Logs tell you what happened. Traces tell you where it happened. You need all three at scale.

🎯 Interview Prep · All Levels

Interview Prep

System design interviews are about demonstrating structured thinking, knowing tradeoffs, and communicating clearly. This section gives you a repeatable framework and walks through 10+ full interview questions.

The 6-Step Framework

Every system design interview should follow this structure. It shows discipline and ensures you don't miss anything important.

1

Clarify Requirements (3–5 min)

Ask questions before designing. What features are in scope? Who are the users? What scale? Read-heavy or write-heavy? What are the latency requirements? Never assume — confirm. Write the requirements on the whiteboard.

2

Estimate Scale (2–3 min)

Back-of-envelope calculations: DAU, requests per second, storage needed, bandwidth. This drives your architecture decisions. Mention if you're estimating QPS, storage, or memory. Don't skip this — it shows engineering sense.

3

High-Level Design (10 min)

Draw the big boxes: client → CDN → load balancer → API servers → cache → databases. Cover the main data flow. Get the interviewer's buy-in before diving deep. Don't detail every component yet.

4

Define the Data Model (5 min)

What are the main entities? What does each table/document look like? What's the primary key? What indexes do you need? What's your read/write pattern? This often reveals your database choice.

5

Deep Dive (15 min)

Pick 2–3 critical/interesting components and go deep. Likely prompted by the interviewer. This is where you show expertise: sharding strategy, caching layer, consistency model, specific algorithms. Discuss tradeoffs.

6

Address Bottlenecks & Failures (5 min)

What are the single points of failure? What happens if the database goes down? What's the hotspot risk? How do you handle a viral moment? Show you can think about operational concerns, not just happy-path design.

Back-of-Envelope Estimation

Interviewers want to see you estimate confidently. These numbers are your vocabulary:

Time Conversions

1 day = 86,400 sec ≈ 100K sec
1 month = 2.5M sec
1 year = 30M sec

Data Sizes

char = 1B · int = 4B · long = 8B
image ≈ 300KB · video/min ≈ 50MB
tweet ≈ 280B

QPS from DAU

If 100M DAU, each makes 5 req/day:
500M req/day ÷ 100K sec = 5000 QPS
Peak ≈ 2–3× = 10–15K QPS

Storage

Twitter: 100M tweets/day
= 100M × 280B ≈ 28GB/day
= ~10TB/year (just text)

Tip: Round aggressively. Say "approximately 10K QPS" not "9,847 QPS." Estimation is about order-of-magnitude thinking, not precision. Justify your assumptions out loud.

Do's & Don'ts

✅ Do

  • Ask clarifying questions first
  • Think out loud — share your reasoning
  • State tradeoffs explicitly
  • Start simple, then evolve
  • Acknowledge what you don't know
  • Draw a diagram (even rough)
  • Check in with interviewer
  • Mention failure scenarios

❌ Don't

  • Jump to coding immediately
  • Choose a DB without justification
  • Design everything in detail first
  • Ignore scale entirely
  • Claim everything is perfect
  • Forget about security
  • Use buzzwords without explaining them
  • Stay silent for too long

Design a URL Shortener (like bit.ly)
BeginnerClassic
+
Requirements

Functional: Given a long URL, return a short URL. Given a short URL, redirect to original. Optional: custom aliases, expiry.

Non-functional: 100M URLs/day, 10:1 read:write ratio, low latency redirect (<10ms), high availability.

Scale Estimation

Writes: 100M/day = ~1200 QPS. Reads: 1.2B/day = 12K QPS. Storage: 100M/day × 500B/URL × 10 years = ~500GB total. Manageable.

Key Design Decisions
  • Hash generation: Use MD5 or SHA256, take first 7 chars (62⁷ ≈ 3.5 trillion combinations). Collision? Append counter and retry.
  • Database: NoSQL (Cassandra or DynamoDB) — simple key-value lookups, horizontally scalable.
  • Caching: Redis cache for hot URLs (80% of reads are on 20% of URLs). Cache with TTL. Reduces DB load dramatically.
  • Redirect: 301 (permanent) = browser caches, less server load. 302 (temporary) = tracks every click, more analytics data.
  • Custom aliases: Allow user to specify shortcode. Check availability in DB first. Use read-your-own-writes consistency.
Data Model

Table: urls(short_code PK, original_url, user_id, created_at, expires_at). Index on original_url if we need reverse lookup (check if long URL already shortened).

Interview Tip

If asked about unique ID generation at scale: use a distributed ID generator (Twitter Snowflake, or pre-generate a pool of IDs). Avoid auto-increment across distributed nodes — they conflict.

Design Instagram (photo sharing)
IntermediatePopular
+
Core Features

Upload photos/videos. Follow users. View news feed. Like and comment. Search.

Key Architecture Decisions
  • Photo storage: Store photos in object storage (S3), not in the database. DB stores metadata (URL, size, user_id, timestamp).
  • CDN: Serve photos from CDN edge nodes. Users around the world get photos from the nearest server. Critical for latency.
  • News feed: Use fanout-on-write for most users (pre-computed feeds in Redis). Use fanout-on-read for users with huge followings (celebrities). Merge at read time.
  • Database: PostgreSQL for users, follows, comments. Cassandra for activity feeds (high write throughput). Both used at Instagram historically.
  • Search: Elasticsearch for hashtag/user search. Not the primary DB.
Bottlenecks to Address
  • Photo upload: async processing — return immediately, process in background (generate thumbnails via Kafka/Lambda)
  • Feed generation: pre-compute vs lazy. Use a hybrid based on follower count
  • Hot content: popular posts get cached aggressively in Redis with high TTL
Design WhatsApp (chat system)
IntermediateClassic
+
Requirements

Functional: 1:1 and group messaging, media (images, video), message delivery status (sent/delivered/read), online presence.

Scale: 2B users, 100B messages/day ≈ 1M QPS.

Key Design Decisions
  • WebSocket vs HTTP polling: Use persistent WebSocket connections for real-time bidirectional communication. HTTP polling is too slow and wasteful. Server can push messages to client instantly.
  • Chat servers: Each client maintains a WebSocket with a chat server. A session service maps user_id → chat_server. Message routing: sender → sender's chat server → look up recipient's chat server → push message.
  • Message storage: HBase (wide-column, high write throughput). Schema: (user_id, conversation_id, timestamp, message_id, content). Time-ordered within conversation.
  • Offline delivery: If recipient is offline, store message in a "pending" queue. On connection, push pending messages. Use message_id to avoid duplicates.
  • End-to-end encryption: Signal Protocol. Keys stored on device only. Server never sees plaintext. Mention this — shows security awareness.
  • Media: Upload to S3, send URL in message. CDN delivers media to recipient. Don't put binary blobs in message DB.
Group Chats

Group chat is fanout problem. For small groups (<100): send individual messages to each member's inbox. For large groups: use Pub/Sub — members subscribe to group topic in Kafka/Redis.

Design Netflix (video streaming)
AdvancedPopular
+
Scale

220M subscribers. 15% of global internet traffic. Need to serve 4K video to millions simultaneously with minimal buffering.

Key Architecture Decisions
  • Video storage & CDN: Raw videos on S3. Transcoded into multiple formats/resolutions (adaptive bitrate) and pushed to Netflix's Open Connect CDN. ISPs host CDN appliances in their networks — video comes from next door, not across the internet.
  • Adaptive Bitrate Streaming (ABR): Client requests chunks of video. If network is slow, it downgrades quality (720p → 480p). If fast, upgrades. Uses DASH or HLS protocol.
  • Video processing pipeline: Original video → Kafka → transcoding workers → multiple quality versions → CDN. Fully async, parallelized.
  • Recommendation engine: Apache Spark for batch ML. Feature store for real-time features. Recommendations precomputed nightly and cached.
  • Metadata service: Microservices for titles, search, user profiles. Cassandra for watch history (time-series, high write).
  • Chaos Engineering: Netflix invented Chaos Monkey — randomly kills production servers to test resilience. Your design should be resilient to this.
Design Uber (ride sharing)
AdvancedClassic
+
Core Problem

Match riders with nearby drivers in real-time. Track driver locations. Handle trip lifecycle. All at global scale, low latency.

Key Design Decisions
  • Location tracking: Drivers send GPS location every 5 seconds via WebSocket/HTTP. This is ~500K updates/sec for Uber's scale. Store in Redis (fast, ephemeral) not a DB. Use geospatial indexing (Redis GEOADD, GEORADIUS commands).
  • Driver matching: When rider requests: query Redis for drivers within radius (geospatial query). Filter by availability. Rank by estimated pickup time. This is the dispatch service — extremely latency-sensitive.
  • Geohashing: Divide the world into a grid of cells (geohash). Drivers in a cell are stored together. Querying nearby drivers = querying current cell + adjacent cells. Efficient and scalable.
  • Trip service: Manages trip lifecycle (requested → accepted → started → completed). Saga pattern for state transitions. Stores in PostgreSQL (ACID needed for financial aspects).
  • Surge pricing: Pub/Sub consumer on trip_requested events. Real-time supply/demand calculation per geohash. Updates pricing for that area.
  • Notifications: Push notifications (APNs/FCM) for match found, driver arriving, etc. Via dedicated notification service fed by Kafka events.
Interview Tip

Geohashing and geospatial queries are the key technical insight here. If you know Redis GEORADIUS and geohash concepts, say so — it shows depth.

Design Twitter (social media feed)
IntermediateClassic
+
The Core Problem

Users post tweets. Users follow each other. When you open Twitter, you see a feed of recent tweets from everyone you follow — in the right order, fast.

Key Design: Feed Generation
  • Naive approach: On feed load, query tweets from all followed users, merge, sort by time. Problem: if you follow 1000 people, that's 1000 DB queries merged in real-time. Too slow.
  • Fanout-on-write (push): When user posts, immediately push tweet_id to all followers' feed lists in Redis. Feed load = single Redis list read. Fast reads, slow writes for popular users (Lady Gaga's tweet fans out to 100M users).
  • Hybrid: Fanout-on-write for users with <1M followers. Fanout-on-read for celebrities. At read time, merge the user's pre-computed feed with real-time queries to celebrities they follow. Twitter uses this approach.
  • Data model: tweets(id, user_id, text, created_at, media_url). follows(follower_id, followee_id). Redis sorted set for feeds: key = user_id, score = timestamp, member = tweet_id.
Design a Web Crawler & Search Engine
AdvancedSenior
+
Crawler Design
  • Seed URLs → URL Frontier: Priority queue of URLs to crawl. Prioritize by PageRank, freshness, domain.
  • Distributed fetchers: Multiple workers fetch URLs in parallel. Respect robots.txt. Rate limit per domain (don't DDoS).
  • Deduplication: Bloom filter to skip already-crawled URLs. Hash content to skip duplicate pages (same content, different URL).
  • Storage: Raw HTML in object storage. Extracted links back to URL frontier. Extracted text to indexing pipeline.
Search Index
  • Inverted index: Maps word → list of (doc_id, positions). "apple" → [(doc1, [5,20]), (doc2, [3])]. Built by MapReduce over crawled pages.
  • Serving: Query → tokenize → look up each word in inverted index → intersect result sets → rank by relevance → return top K.
  • Ranking: PageRank (link graph), TF-IDF (term frequency), freshness, click-through rate, semantic similarity.
Design YouTube (video platform)
Intermediate
+
Upload Flow

Client uploads video → API server streams to raw storage (S3) → publishes "video uploaded" event to Kafka → Transcoding workers consume event → generate multiple resolutions → push to CDN → update video metadata DB to status=available → notify uploader.

Key Decisions
  • Resumable uploads: Large video files need resumable upload. Upload in chunks. Track progress server-side. Resume from last chunk on failure.
  • Transcoding: Computationally expensive, async. Encode to H.264/H.265 at 1080p, 720p, 480p, 360p. Generate thumbnails. All in parallel with multiple workers.
  • View count: Don't update DB on every view — too much write load. Batch updates every minute using Redis counter. Consistency can be eventual here.
  • Comments: Nested comments — use closure table or materialized path pattern. Or denormalize top-level and reply structure separately.
Design a Rate Limiter
BeginnerComponent
+
Design
  • Algorithm: Token bucket (most practical). Each user gets a bucket of tokens (e.g., 100 req/min). Each request consumes a token. Tokens refill at a fixed rate. Bucket size = max burst.
  • Storage: Redis. Key: rate_limit:{user_id}. Value: {tokens, last_refill_time}. Atomic update via Lua script (check + update = one atomic operation, prevents race conditions).
  • Response: If token available → consume token → allow request. Else → return 429 with Retry-After header = time until next token.
  • Distributed: Multiple API servers all share the same Redis instance. Rate limit enforced globally, not per-server.
  • Config: Rate limits per API key, per user tier. Store limits in a config DB (so you can change without deployment).
Design a Distributed Key-Value Store
AdvancedSenior
+
Design (like a mini DynamoDB)
  • Data partitioning: Consistent hashing ring. Keys distributed across nodes. Virtual nodes prevent uneven distribution. Adding/removing nodes only moves ~1/N keys.
  • Replication: Each key replicated to N nodes (next N clockwise on the ring). Tunable: N=3, W=2, R=2 for quorum (W+R > N = strong consistency).
  • Consistency: Choose: quorum (strong) or last-write-wins (eventual). Vector clocks for conflict detection. Application-level conflict resolution for true eventual consistency.
  • Failure handling: Gossip protocol for node health. If a node is down, use sloppy quorum + hinted handoff (another node temporarily holds the data, hands it off when original recovers).
  • Storage engine: SSTable + LSM Tree (Log-Structured Merge Tree). Writes go to in-memory memtable first (fast). Flushed to sorted SSTables on disk. Reads merge SSTables. Used by Cassandra, LevelDB, RocksDB.
