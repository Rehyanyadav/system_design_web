CAP Theorem Quick Guide
PickMeaningExamples
CPConsistent + partition tolerant. Returns error if can't guarantee fresh data.HBase, etcd, Zookeeper
APAvailable + partition tolerant. Returns possibly stale data during partition.Cassandra, DynamoDB, CouchDB
CANot real in distributed systems — partitions always happen.Single-node SQL

Real choice: C vs A during a partition (P is always required).

Cache Eviction
LRU — Least Recently UsedMost common. Evict oldest access.
LFU — Least Frequently UsedEvict accessed fewest times.
TTL — Time To LiveExpire after fixed duration.
FIFO — First In First OutEvict oldest insertion.

80/20 rule: 20% of data = 80% of requests. Cache that 20%.

Advanced Patterns at a Glance
Distributed System Patterns
  • Circuit Breaker — Stop calling failing service. 3 states: closed/open/half-open.
  • Saga Pattern — Distributed transactions via compensating actions. Choreography or orchestration.
  • Event Sourcing — Store events, not state. Replay to reconstruct. Full audit log.
  • CQRS — Separate read and write models. Optimize each independently.
  • Bulkhead — Isolate failures. Separate thread pools per downstream service.
  • Sidecar — Deploy infra logic (logging, mesh) alongside app without modifying app code.

Interview Answer Framework
The 6 Steps — Every Interview
1
Clarify
Ask scope, scale, constraints. Never assume.
2
Estimate
QPS, storage, bandwidth. Back-of-envelope.
3
High-Level
Big boxes: LB, servers, cache, DB. Main flow.
4
Data Model
Entities, schema, access patterns.
5
Deep Dive
2-3 interesting components. Tradeoffs.
6
Failures
SPOFs, hotspots, what if X goes down?

Red Flags Interviewers Watch For
  • Jumping to solution without clarifying requirements
  • Picking databases/tech without justification
  • No mention of scale or failure scenarios
  • Designing a single point of failure and not catching it
  • Silence for more than 30 seconds without communicating
  • Overengineering a simple problem
