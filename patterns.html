Design Patterns

This is where it gets interesting. Patterns are the solutions to problems that appear in every system. Understanding them — and when to use them — is what interviewers are testing.

Sharding (Horizontal Partitioning)

Sharding splits a large dataset across multiple database nodes (shards). Each shard holds a subset of the data. When no single database can hold all your data or handle all your queries, you shard.

Key question: What is the shard key? The shard key determines how data is distributed. A bad shard key creates hotspots (all traffic goes to one shard).

Sharding Strategies

Hash-Based Sharding

Hash the shard key and take modulo number of shards. Even distribution. Problem: adding a shard requires rehashing everything.

Range-Based Sharding

Split by key ranges (A–M on shard 1, N–Z on shard 2). Good for range queries. Risk of hotspots if data isn't uniformly distributed.

Directory-Based Sharding

Lookup table maps keys to shards. Very flexible. Single lookup table becomes a bottleneck and single point of failure.

Geographic Sharding

Data partitioned by region (US shard, EU shard, APAC shard). Great for data residency, latency. Imbalanced if regions have different loads.

Sharding gotcha: Cross-shard queries (JOINs across shards) are expensive or impossible. You must design your access patterns so most queries hit a single shard.

Consistent Hashing

Regular hash-based sharding has a problem: when you add or remove a node, you have to remap almost all keys. Consistent hashing solves this.

How it works

Imagine a ring of hash values from 0 to 2³². Servers and keys are both placed on this ring by hashing their identifiers. A key belongs to the first server clockwise from it on the ring.

When you add a server: only the keys between the new server and its predecessor move. That's ~1/N of total keys, not all of them.

When you remove a server: only its keys move to the next server clockwise. Again, ~1/N keys affected.

Real World Use

Where consistent hashing is used

Amazon DynamoDB, Apache Cassandra, Vimeo's memcache layer, Discord's server routing. It's the standard approach for distributing load across a dynamic set of nodes.

Virtual nodes: To prevent load imbalance (some servers getting more of the ring than others), each physical server is assigned multiple virtual positions on the ring. Cassandra uses 256 virtual nodes per server by default.

Replication

Replication keeps copies of data on multiple nodes. This provides fault tolerance (if one dies, others serve) and read scaling (distribute reads across replicas).

Leader-Follower (Master-Replica)

One leader accepts all writes. Followers replicate and handle reads. Most common pattern.

✅ Simple, strong consistency for writes ❌ Write bottleneck is the leader

Multi-Leader

Multiple leaders accept writes (often one per datacenter). Great for geo-distributed systems. Hard problem: write conflicts between leaders.

Leaderless (Quorum-Based)

Any node can accept writes. Reads/writes require a quorum (W + R > N where N=replicas). Used by Cassandra and DynamoDB.

🎯 Interview Tip "If your leader goes down, how does the system recover?" — Know the answer: automatic leader election (Raft/Paxos), or manual failover. In practice: PostgreSQL uses pg_ha or Patroni; MySQL uses Orchestrator. Cloud managed DBs handle it for you.

CAP Theorem

CAP Theorem states that a distributed system can only guarantee two of three properties simultaneously:

C

Consistency

Every read gets the most recent write, or an error

A

Availability

Every request gets a (possibly stale) response — no errors

P

Partition Tolerance

System continues despite network partition between nodes

The real choice: In a real distributed system, network partitions WILL happen. So you can't give up P. The real decision is: during a partition, do you prefer Consistency (reject requests) or Availability (serve potentially stale data)?

CP systems (choose consistency): HBase, Zookeeper, etcd. Stop serving or return errors during partition.
AP systems (choose availability): Cassandra, DynamoDB, CouchDB. Serve stale data during partition, reconcile after.

Real Example · Amazon DynamoDB

The AP choice — and why Amazon made it

DynamoDB is AP: it will serve reads even if they might be stale. Amazon's reasoning: a customer seeing a slightly stale shopping cart is far less bad than their cart being unavailable. For financial transactions, they add strongly consistent reads at higher cost. Context determines the right tradeoff.

ACID vs BASE

These are two philosophical approaches to database design that map to SQL and NoSQL respectively.

ACID

  • Strong consistency
  • Predictable behavior
  • Harder to scale
  • Higher latency under load

PostgreSQL, MySQL, Oracle

BASE

Basically Available, Soft state, Eventually consistent

  • Highly scalable
  • High availability
  • Eventual (not immediate) consistency
  • App must handle stale data

Cassandra, MongoDB (default), DynamoDB

Rate Limiting

Rate limiting prevents any single user or service from overwhelming your system. It's essential for public APIs and high-scale services.

Algorithms

Token Bucket

Bucket fills with tokens at a fixed rate. Each request consumes a token. Allows bursts up to bucket size. Used by AWS, Stripe.

Leaky Bucket

Requests enter a queue (bucket) and are processed at a fixed rate. Smooths out bursts. If bucket is full, requests are dropped.

Fixed Window Counter

Count requests in a fixed time window (e.g., 100 req/minute). Simple but has edge case: 200 requests at window boundary.

Sliding Window Log

Track timestamps of requests. Remove old timestamps. Count remaining. Accurate but memory-intensive at scale.

Implementation: Store rate limit state in Redis (per user key). Token bucket is most popular in practice. Return 429 Too Many Requests with Retry-After header when limit is hit.

Circuit Breaker Pattern

When a service is down or slow, repeatedly trying to call it wastes resources and creates cascading failures. The circuit breaker stops calls to a failing service and allows it to recover.

Three States

Closed — normal operation, all requests pass through. Count failures.
Open — failure threshold exceeded. All requests fail fast (no call made to downstream). A timer starts.
Half-Open — timer expires. Allow one test request. Success → close the circuit. Failure → stay open.

Real Example · Netflix Hystrix

Netflix's circuit breaker in practice

Netflix wraps every service call in a circuit breaker (Hystrix library). If the recommendation service is down, the circuit opens and Netflix serves a degraded response (top 10 popular movies) instead of an error. Users see something rather than nothing — this is graceful degradation.

The Fanout Problem

Fanout is when one write operation causes many downstream operations. Classic example: when a celebrity posts on Twitter, their tweet must be delivered to 100M followers.

Fanout on Write (Push)

  • Fast reads (pre-computed feeds)
  • Slow writes for popular users
  • Storage cost (100M copies of tweet)

Used for: most users with moderate followers

Fanout on Read (Pull)

  • Fast writes (one write)
  • Slow reads (aggregate at read time)
  • Higher read-time compute

Used for: celebrities/users with huge followings

Twitter's hybrid approach: Push model for normal users. Pull model for users with >1M followers (Lady Gaga, Obama). At read time, merge the push feed with pull from celebrities. This is a classic interview answer.

Bloom Filters

A Bloom filter is a probabilistic data structure that answers "is this element in the set?" with zero false negatives but possible false positives. Uses very little memory.

Use cases: check if a username is taken before querying the database. Check if a URL has been crawled. Check if a password is in a breach list. The Bloom filter gives a fast "definitely not" or "probably yes" answer.

Rule: Bloom filter says NO → definitely not in the set (no DB query needed). Bloom filter says YES → probably in set, but query DB to confirm. This eliminates most unnecessary DB queries.
